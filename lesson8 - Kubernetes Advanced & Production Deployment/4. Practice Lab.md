# Practice Lab - Lesson 8: Kubernetes Core Workloads

## Setup: Create and Switch to Practice Namespace

Before we begin, let's create a dedicated namespace for this practice lab to keep our resources organized and isolated.

1. **Create the namespace:**
   ```bash
   kubectl create namespace lesson8
   ```

2. **Switch to the namespace:**
   ```bash
   kubectl config set-context --current --namespace=lesson8
   ```
   
   Alternatively, if you have `kubens` installed:
   ```bash
   kubens lesson8
   ```

3. **Verify you're in the correct namespace:**
   ```bash
   kubectl config view --minify | grep namespace
   ```
   You should see `namespace: lesson8`

Now all the resources we create will be in the `lesson8` namespace!

## 1. The Core: Deployment & ReplicaSet

**Concepts:**
* A **Deployment** is a "desired state" for your app. You tell it, "I want 3 replicas of this app image."
* The Deployment automatically creates a **ReplicaSet**.
* The **ReplicaSet**'s job is to make reality match the desired state. If a pod crashes, the ReplicaSet brings up a new one.

You almost **never** create a ReplicaSet directly; you always use a Deployment.

### Practice

1.  Create a file named `deployment.yaml`:

    ```yaml
    # deployment.yaml
    apiVersion: apps/v1
    kind: Deployment
    metadata:
      name: my-web-app
    spec:
      replicas: 2 # Tell the Deployment we want 2 pods
      selector:
        matchLabels:
          app: web # This label connects the Deployment to its pods
      template: # This is the "cookie cutter" for the pods
        metadata:
          labels:
            app: web # The pod's label
        spec:
          containers:
          - name: nginx-container
            image: nginx:1.25 # A simple web server
            ports:
            - containerPort: 80
    ```

2.  **Apply the YAML:**
    ```bash
    kubectl apply -f deployment.yaml
    ```

3.  **Check your work:**
    ```bash
    # Check the Deployment status
    kubectl get deployments
    
    # Check the ReplicaSet it created
    kubectl get replicasets
    
    # Check the pods it created
    kubectl get pods
    ```
    You should see `my-web-app` with 2/2 replicas ready and two pods running.

4.  **Practice restarting the deployment:**
    
    Sometimes you need to restart your deployment (for example, to pick up new configuration or force a pod refresh). Here are several ways to do it:
    
    ```bash
    # Method 1: Restart rollout (Kubernetes 1.15+) - RECOMMENDED
    kubectl rollout restart deployment my-web-app
    
    # Method 2: Scale down and up
    kubectl scale deployment my-web-app --replicas=0
    kubectl scale deployment my-web-app --replicas=2
    
    # Method 3: Delete pods (ReplicaSet will recreate them)
    kubectl delete pods -l app=web
    ```
    
    **Watch the restart process:**
    ```bash
    # Watch the rollout status
    kubectl rollout status deployment my-web-app
    
    # Watch pods being recreated in real-time
    kubectl get pods -w
    ```
    
    **View rollout history:**
    ```bash
    # See deployment rollout history
    kubectl rollout history deployment my-web-app
    ```

## 2. Exposing the App: Service & Ingress


### Practice

1.  **Ensure an Ingress Controller is Running:** An Ingress resource won't work without an Ingress controller.
    
    To check, you can run:
    ```bash
    kubectl get pods -n ingress-nginx # (Or the namespace for your controller)
    ```

    **For kind clusters:** If you're using kind, the ingress controller may run in the default namespace. Check with:
    ```bash
    kubectl get ingressclass
    ```

2.  Create a file named `service.yaml`. This creates the internal front door.

    ```yaml
    # service.yaml
    apiVersion: v1
    kind: Service
    metadata:
      name: my-web-service
    spec:
      type: ClusterIP # Only reachable inside the cluster (perfect for Ingress)
      selector:
        app: web # This MUST match the labels on your pods!
      ports:
      - protocol: TCP
        port: 80 # The Service's port
        targetPort: 80 # The port on the pods (from our deployment.yaml)
    ```

3.  Create a file named `ingress.yaml`. This is the external router.

    ```yaml
    # ingress.yaml
    apiVersion: networking.k8s.io/v1
    kind: Ingress
    metadata:
      name: my-web-ingress
    spec:
      ingressClassName: nginx # IMPORTANT: Specify the ingress class
      rules:
      - http:
          paths:
          - path: /
            pathType: Prefix
            backend:
              service:
                name: my-web-service # Points to the Service we just made
                port:
                  number: 80
    ```

4.  **Apply the YAML files:**
    ```bash
    kubectl apply -f service.yaml
    kubectl apply -f ingress.yaml
    ```

5.  **Test it:**
  - Access the test app using: http://<ec2-ip>:8080/

## 3. Adding Storage: PV & PVC

Your NGINX pods are stateless. Let's add persistent storage.

* **PersistentVolume (PV):** A piece of storage in the cluster (like a hard drive).
* **PersistentVolumeClaim (PVC):** A *request* for storage by an app. Kubernetes matches the PVC request to an available PV.

### Practice

1.  Create `pv.yaml`. We'll use `hostPath`, which is **not for production** but is simple for single-node testing. It maps to a folder *on the cluster node itself*.

    ```yaml
    # pv.yaml
    apiVersion: v1
    kind: PersistentVolume
    metadata:
      name: my-app-pv
    spec:
      capacity:
        storage: 1Gi
      accessModes:
        - ReadWriteOnce # Can be mounted by one node at a time
      hostPath:
        path: "/data/my-app" # This path MUST exist on your cluster's node
    ```

2.  Create `pvc.yaml`. This is our app's *request*.

    ```yaml
    # pvc.yaml
    apiVersion: v1
    kind: PersistentVolumeClaim
    metadata:
      name: my-app-pvc
    spec:
      accessModes:
        - ReadWriteOnce
      resources:
        requests:
          storage: 1Gi # Requests 1GB. This will match our PV!
      storageClassName: "" # Use manual PV binding (required for hostPath PVs)
    ```

3.  **Apply them:**
    ```bash
    kubectl apply -f pv.yaml
    kubectl apply -f pvc.yaml
    ```
    * Check status: `kubectl get pv,pvc`. Wait until you see the status of both as `Bound`.

4.  **Update `deployment.yaml`** to use the PVC.

    ```yaml
    # deployment.yaml (Updated)
    apiVersion: apps/v1
    kind: Deployment
    metadata:
      name: my-web-app
    spec:
      replicas: 2
      selector:
        matchLabels:
          app: web
      template:
        metadata:
          labels:
            app: web
        spec:
          # --- ADD THIS SECTION ---
          volumes:
          - name: web-storage
            persistentVolumeClaim:
              claimName: my-app-pvc # Use the PVC we created
          # --- END ADD ---
          containers:
          - name: nginx-container
            image: nginx:1.25
            ports:
            - containerPort: 80
            # --- ADD THIS SECTION ---
            volumeMounts:
            - name: web-storage # Matches the 'volumes' name above
              mountPath: /usr/share/nginx/html # Mount it over NGINX's default web root
            # --- END ADD ---
    ```

5.  **Re-apply the Deployment:**
    ```bash
    kubectl apply -f deployment.yaml
    ```
    Kubernetes will perform a "rolling update" to attach the storage.

6.  **Test the persistence:**
    * Get a pod name: `kubectl get pods`
    * `exec` into one of the pods (replace with your pod name):
        ```bash
        kubectl exec -it my-web-app-xxxx-yyyy -- /bin/bash -c 'echo "Hello from persistent storage!" > /usr/share/nginx/html/index.html'
        ```
    * Test your service with the new content:
        ```bash
        kubectl run test-pod --image=curlimages/curl --rm -it --restart=Never -- curl my-web-service
        ```
        You should see "Hello from persistent storage!"
    * **The real test:** Delete the pod you exec'd into:
        ```bash
        kubectl delete pod my-web-app-xxxx-yyyy
        ```
    * Wait a few seconds. The ReplicaSet creates a *new* pod.
    * Test your service again:
        ```bash
        kubectl run test-pod --image=curlimages/curl --rm -it --restart=Never -- curl my-web-service
        ```
        You will *still* see "Hello from persistent storage!" The data survived.

## 4. Running Tasks: Job & CronJob

Sometimes you just want to run a task *once* (like a database migration) or on a *schedule* (like a nightly backup).

* **Job:** Creates one or more pods and ensures they run to completion (exit 0).
* **CronJob:** A Job on a schedule (like a Linux cron).

### Practice

1.  Create `job.yaml` for a one-time task.

    ```yaml
    # job.yaml
    apiVersion: batch/v1
    kind: Job
    metadata:
      name: my-one-time-job
    spec:
      template:
        spec:
          containers:
          - name: pi-calculator
            image: alpine:3.19 # Using a lightweight image for faster startup
            command: ["/bin/sh", "-c", "echo 'Job completed! Pi is approximately 3.14159265358979323846' && sleep 2"]
          restartPolicy: Never # 'Never' or 'OnFailure' are valid for Jobs
    ```

2.  **Apply the Job:**
    ```bash
    kubectl apply -f job.yaml
    ```

3.  **Check its progress:**
    ```bash
    # Wait for the job to complete
    kubectl get jobs
    
    # Get the pod created by the job
    kubectl get pods
    
    # Check the logs of the completed pod (replace with your pod name)
    kubectl logs my-one-time-job-xxxx
    ```
    You'll see the job completion message in the logs!

4.  Create `cronjob.yaml` for a scheduled task.

    ```yaml
    # cronjob.yaml
    apiVersion: batch/v1
    kind: CronJob
    metadata:
      name: my-nightly-job
    spec:
      schedule: "*/1 * * * *" # This means "run every 1 minute" for easy testing
      jobTemplate: # This is just a Job template, like the one above
        spec:
          template:
            spec:
              containers:
              - name: hello-logger
                image: busybox:1.36
                command: ["/bin/sh", "-c", "echo 'Hello from the cron job!'"]
              restartPolicy: OnFailure
    ```

5.  **Apply the CronJob:**
    ```bash
    kubectl apply -f cronjob.yaml
    ```

6.  **Watch it run:**
    ```bash
    # Watch the CronJob
    kubectl get cronjobs
    
    # Watch new Jobs being created every minute (use -w to watch)
    kubectl get jobs -w
    ```
    You'll see new jobs being created every minute, like `my-nightly-job-28594001`, `my-nightly-job-28594002`, etc. You can check their logs just like you did with the regular Job.

## 🧹 Cleanup

To delete all the resources you created from your cluster:

```bash
# Delete all resources from your YAML files
# Run this from the directory containing all your .yaml files
kubectl delete -f .

# Optional: Delete the entire namespace (this will remove ALL resources in the namespace)
kubectl delete namespace lesson8

# Switch back to the default namespace
kubectl config set-context --current --namespace=default
```

> **Note:** Deleting the namespace will remove all resources at once, which is often the fastest cleanup method for practice labs.
